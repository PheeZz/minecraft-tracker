# Задачи по сотам (Bee comb-requirement tasks) — дизайн

Дата: 2026-05-31. Раздел: трекер пчёл (`src/trackers/bees`).

## Проблема и цель

В модпаке некоторые предметы крафтятся из **нескольких разных сот одновременно**
(например, предмет требует фруктовые + каучуковые + медовые соты). Какой предмет
какие соты требует — заранее неизвестно, это знает только игрок.

Нужен лёгкий «постановщик задач»: игрок заводит задачу (= цель-предмет), вписывает
в неё нужные соты, и приложение **само показывает прогресс** — какие из этих сот он
уже может производить, а для недостающих подсказывает, какую пчелу проще вывести.

Фича опирается на две уже существующие связи:

- **сота → пчёлы-производители** — индекс `COMBS` (`domain/combs.ts`);
- **пчела → есть ли на складе** — инвентарь (`useBeesStore.have`).

Сложение даёт цепочку: **задача → нужные соты → пчёлы → есть ли они у тебя**.

## Решения, принятые на брейншторме

1. **Что значит «сота закрыта» — Вариант А (авто из инвентаря).** Сота считается
   закрытой, как только на складе есть хотя бы одна пчела, которая её производит.
   Никаких ручных галочек и количеств. Прогресс задачи = доля закрытых сот.
2. **Задача содержит только соты** (плюс название). Не пчёл, не обычные предметы,
   не количества.
3. **Задачи создаёт и редактирует сам игрок** в приложении.
4. **Размещение — внутри раздела «Пчёлы».** Задачи открываются **большой модалкой**
   поверх текущего экрана (не полноэкранным режимом — это было бы избыточно).
5. **Навигация чинится попутно**: маленькая спрятанная кнопка «инвентарь» в левой
   колонке убирается; вводится заметная полоса-переключатель сверху.

## Модель данных

```ts
/** Задача: цель-предмет, требующая набор сот. domain/tasks.ts */
export interface BeeTask {
  id: string // crypto.randomUUID()
  name: string // имя предмета/цели, задаёт игрок
  combs: string[] // имена требуемых сот (ключи COMBS), без дублей
  collapsed?: boolean // свёрнута ли карточка (готовые удобно сворачивать)
}
```

Хранение: один ключ `bees.tasks` → `BeeTask[]` через обёртку `storage`
(`@/shared/persistence/storage`). Без миграций. `id` — `crypto.randomUUID()`.

### Статус соты (производная, не хранится)

Для имени соты `C`, при текущем складе:

```ts
type CombState = 'have' | 'ready' | 'todo'
interface CombStatus {
  comb: string
  state: CombState
  bee: string | null // какая пчела закрывает/подсказана
  depth: number // шагов выведения подсказанной пчелы (0 для have/ready)
}
```

Алгоритм (использует существующий `store.producersOf(comb)` — уже отсортирован по
глубине ↑, затем шансу ↓; `RankedProducer` несёт `depth`):

1. `producers = producersOf(comb)`.
2. Если среди них есть `isHave(p.bee)` → `state='have'`, `bee` = эта пчела
   (берём с наибольшим шансом среди имеющихся), `depth=0`.
3. Иначе `easiest = producers[0]`:
   - `easiest.depth === 0` → `state='ready'` (можно вывести прямо сейчас);
   - `easiest.depth > 0` → `state='todo'` (нужно вывести, `depth` шагов);
   - `bee = easiest.bee`.
4. Защита: если у соты нет производителей (`producers` пуст) → `state='todo'`,
   `bee=null`, `depth=0` (в норме не случается, т.к. соты выбираются из `COMB_NAMES`).

### Прогресс задачи (производная)

`taskProgress(task) = { done: число сот со state==='have', total: task.combs.length }`.
Задача «готова», когда `done === total` и `total > 0`.

Все производные считаются в реактивном scope (computed/шаблон) и сами
пересчитываются при изменении склада — так же, как чипы инвентаря: они читают
`have`/`depthOf` и потому отслеживают зависимость.

## Изменения в сторе (`useBeesStore.ts`)

Навигация (заменяет нынешний `inventoryOpen`/`toggleInventory`):

- `view: ref<'graph' | 'inventory'>` (по умолчанию `'graph'`), `setView(v)`.
- `tasksOpen: ref<boolean>` (модалка задач, независима от `view`), `toggleTasks()` /
  `openTasks()` / `closeTasks()`.

Задачи:

- `tasks: ref<BeeTask[]>` (загрузка из `storage.get('bees.tasks', [])`).
- `addTask(name, combs)` — создаёт `{id: randomUUID, name, combs: dedupe(combs)}`,
  пушит, сохраняет.
- `updateTask(id, patch)` — меняет name/combs, сохраняет.
- `removeTask(id)` — удаляет, сохраняет.
- `toggleTaskCollapsed(id)` — переключает `collapsed`, сохраняет.
- `persistTasks()` — единичная запись `storage.set('bees.tasks', tasks.value)`.

Производные статусов сот живут в `domain/tasks.ts` как чистые функции
`combStatus(comb, have, producersOf)` и `taskProgress(task, ...)`, чтобы их можно
было юнит-тестировать без Pinia. Компонент вызывает их через стор.

Существующие экспорты, используемые `BeeRail`/`BeePanel`/`BeesView`, сохраняются;
`inventoryOpen`/`toggleInventory` удаляются, их потребители переводятся на `view`.

## UI

### Полоса-переключатель (новое, в `BeesView.vue`)

Тонкая полоса сверху всей области пчёл, видна во всех режимах:

```
[ 🌿 Граф ] [ 📦 Инвентарь ]                         [ ✅ Задачи ]
```

- Слева — сегментированный переключатель `view`: **Граф / Инвентарь**
  (заодно решает проблему спрятанной кнопки инвентаря).
- Справа — кнопка **«Задачи»** (открывает модалку; `tasksOpen`). На кнопке —
  счётчик незакрытых задач (например, бейдж «2»), если есть.
- «Граф» — нынешняя 3-колоночная раскладка (рейл/граф/панель). «Инвентарь» —
  полноэкранный экран (как сейчас). Модалка задач рисуется поверх любого из них.
- Из `BeeRail` убирается кнопка «инвентарь» (строка `invbar`); «только мои»,
  счётчик склада и «сбросить» остаются (они про список графа).

### Модалка «Задачи» (`BeeTasksModal.vue`)

Центрированное окно ~900×600 (адаптив: на узких экранах — почти на весь экран),
затемнённый backdrop, скролл внутри. Содержимое — список задач карточками:

```
┌─ Задачи ──────────────────────────────────────────  ✕ ─┐
│  ➕ Новая задача                                         │
│ ┌──────────────────────────────────────────────────┐   │
│ │ Предмет №1                             2 / 3  ✎ 🗑 │   │
│ │ ▓▓▓▓▓▓▓░░░  67%                                    │   │
│ │  ✅ Медовые соты — есть (пчела «Лесная»)            │   │
│ │  🟡 Фруктовые — готова, выведи «Z»       [в граф ▸]│   │
│ │  ❌ Каучуковые — нужна «Y», 2 шага       [в граф ▸]│   │
│ └──────────────────────────────────────────────────┘   │
│ ┌──────────────────────────────────────────────────┐   │
│ │ Улучш. схема  ✅ готово                  4 / 4     │   │  ← свёрнута
│ └──────────────────────────────────────────────────┘   │
└─────────────────────────────────────────────────────────┘
```

Карточка задачи (`BeeTaskCard.vue`):

- заголовок (название) + прогресс `done/total` + кнопки **✎ изменить** / **🗑 удалить**;
- прогресс-полоска (доля закрытых сот);
- список нужных сот, у каждой иконка соты (`CombIcon`) и статус:
  - ✅ **есть** — `state==='have'`, подпись «есть (пчела «…»)»;
  - 🟡 **готова** — `state==='ready'`, «готова, выведи «…»» + кнопка «в граф ▸»;
  - ❌ **нужно вывести** — `state==='todo'`, «нужна «…», N шагов» + кнопка «в граф ▸»;
- готовые задачи (`done===total`) по умолчанию свёрнуты (только заголовок), клик
  по заголовку разворачивает (`toggleTaskCollapsed`).

Создание/редактирование (`BeeTaskEditor.vue`) — **инлайн внутри модалки** (без
модалки-в-модалке): клик по «➕ Новая задача» разворачивает форму вверху списка;
клик по ✎ на карточке открывает ту же форму, предзаполненную, на месте карточки.
Форма:

- поле названия;
- выбор сот: список `COMB_NAMES` с поиском и чекбоксами (стиль — как в рейле/
  инвентаре, переиспользуем `CombIcon` и `shortComb`);
- «Сохранить» доступно при непустом названии и ≥1 соте; дубли сот отсекаются.
  «Отмена» сворачивает форму без изменений.

Кнопка **«в граф ▸»** у соты (показывается только при `state!=='have'` и
`bee!==null`): `closeTasks()` → `setView('graph')` → `selectBee(status.bee)`
(строит дерево выведения этой пчелы в `BeeChainGraph`). Так из задачи попадаешь
сразу к плану «кого скрестить». Для соты без производителя (`bee===null`) кнопки
нет — только подпись «нет известного производителя».

### Доступность / поведение модалки

- `role="dialog"`, `aria-modal="true"`, `aria-labelledby` на заголовок.
- Закрытие: крестик, клик по backdrop, `Esc`. Фокус-трап внутри; при открытии фокус
  на первый интерактив, при закрытии — возврат на кнопку «Задачи».
- Анимация появления (opacity/transform), уважает `prefers-reduced-motion`
  (глобальный блок в `style.css`).

## Тестирование

- Юнит (`tasks.spec.ts`): `combStatus` для всех трёх состояний (have/ready/todo) при
  разных складах; `taskProgress`; дедуп сот; пустой/без-производителей кейс.
- Playwright (Chromium + WebKit): открыть модалку из полосы; создать задачу с 2–3
  сотами; убедиться, что статусы и прогресс верны; отметить пчелу в инвентаре →
  прогресс задачи вырос реактивно; «в граф ▸» закрывает модалку и строит дерево
  нужной пчелы; редактирование/удаление; пустое состояние; `Esc`/backdrop закрывают.
- Прогон существующих 87 юнитов + typecheck + eslint без ошибок.

## Вне рамок (YAGNI)

- Количества сот / счётчики предметов.
- Требования, отличные от сот (пчёлы, обычные предметы).
- Ручной оверрайд статуса соты (доверяем авто-логике; можно добавить позже).
- Сортировка/drag задач, категории, импорт/экспорт.

## Состояния и крайние случаи

- Нет задач → пустое состояние в модалке («Пока нет задач — создай первую»).
- Задача без сот не сохраняется (требуем ≥1).
- Дубли сот в задаче отсекаются при сохранении.
- Сота без производителей → `state='todo'`, `bee=null` (подпись «нет известного
  производителя»); защита от пустого `producersOf`.
- Бейдж на кнопке «Задачи» = число незакрытых задач (`done<total`); 0 → без бейджа.
